**One-line description (≤25 words):**
One citizen report — photo + location — routed in parallel to every government
department that must act, with database-enforced privacy and a public
accountability dashboard.

**The problem (≤100 words):**
Citizens don't know which department owns their problem. Existing channels —
municipal apps, grievance portals, helplines — are single-authority: a
complaint reaches one office, and incidents spanning departments (a fallen
electric pole blocking a road; a fire needing ambulance cover) die in
forwarding chains. Officials lack a unified, accountable view; citizens get
"disposed" complaints rather than resolved ones. In emergencies, wrong routing
costs minutes.

**The solution (≤150 words):**
JanVyuha lets a citizen file one report — photo/video + map location, in
Telugu, Hindi, Tamil or English, by voice if preferred — and routes it
simultaneously to every department that must act (Fire, Police, Ambulance,
Municipal, Electricity, Water, Animal Welfare). Each department gets its own
queue and independent status timeline; the citizen tracks all of them on one
screen, receives push/email updates, rates the resolution, and can reopen a
report closed without real resolution. Department isolation is enforced in
the database (Row-Level Security), not just the interface; evidence photos
are private; every administrative action is audit-logged. A public
transparency dashboard publishes anonymised, department-wise resolution
performance with open-data export. AI (photo triage, translation) assists but
never gates. Works offline on low-end Android as an installable web app.
Designed to DPDP Act 2023.

**What makes it different (≤100 words):**
Parallel multi-department routing with per-department status does not exist
in any deployed Indian system — CPGRAMS, Swachhata, municipal apps and DIGIT
PGR are all single-authority-per-complaint. JanVyuha also pairs civic
reporting with emergency-adjacent categories under severity-based SLAs
(15-minute acknowledge for critical), and publishes a department performance
leaderboard — the "consequence architecture" the CPGRAMS literature says is
missing. Full-stack Telugu/Hindi/Tamil/English including department names and
voice input.

**Stage / traction (≤100 words):**
Fully working product, live at [prod URL], built and deployed: 7 departments,
4 languages (679+ translated keys each), offline reporting, Web Push + email
notifications, admin console, SLA analytics, public open-data dashboard.
Engineering quality: typed codebase, 57 automated tests, CI on every change.
Zero-cost pilot configuration on managed cloud (India region). Seeking a
supervised one-district pilot with a nodal officer; the authority needs no
budget, procurement, or hardware to evaluate. Costed scale-up plan prepared.

## 3 · Sequencing & follow-through

1. **Day 1:** TSIC innovator registration → note the id → send the cinno@
   email the same day. Submit the T-Hub government contact form.
2. **Day 1–2:** Send the Telangana outreach letters (ITE&C secretary desk,
   minister's office, GHMC) — the runbook in `pre-outreach-checklist.md`.
   The two tracks reinforce each other: if the secretary's office asks TSIC
   about you, being registered is a credibility mark.
3. **Week 1:** If in/near Hyderabad, ask TSIC for a 15-minute in-person demo
   slot at T-Hub 2.0 — a live phone demo lands far better than a link.
4. **Day 8–10:** Follow up (one email each) if silent; TSIC's phone line is
   answerable — call once after the follow-up, not before.
5. **Log everything** in the send log alongside the ministry letters.

**Consistency rules across both applications and all letters:**
- Same one-liner everywhere (the ≤25-word description above).
- Same ask everywhere (supervised one-district pilot, 8–12 weeks, nodal
  officer, no cost/procurement to the department).
- No "free/₹0" framing; "no budget, procurement, or hardware needed to
  evaluate" + costed scale-up plan.
- Same disclosure: independent proposal, not an official service, does not
  replace 112.
